## C. Glossar (Ergänzung)

| Begriff                            | Bedeutung                                                                                                                                                                                                                                     |
|------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| API                                | Application Programming Interface: Schnittstelle, die es verschiedenen Softwarekomponenten ermöglicht, miteinander zu kommunizieren und Daten auszutauschen.                                                                                  |
| Beta‑Test / Beta‑Release           | Vorabveröffentlichung einer App‑Version an eine eingeschränkte Testergruppe, um Stabilität und Usability in realen Nutzungsszenarien zu prüfen und Feedback zu sammeln.                                                                       |
| Burn (Kosten‑Burn)                 | Kumulativer Verbrauch von Budget oder Personentagen über die Zeit; dient als Indikator für den finanziellen Fortschritt gegenüber dem Plan.                                                                                                   |
| CI/CD-Pipeline                     | Automatisierte Kette zur kontinuierlichen Integration (Build, Lint, Tests) und Auslieferung von Software, die bei Änderungen am Code ausgeführt wird.                                                                                         |
| Deliverable                        | Konkretes Ergebnis oder Artefakt, das im Rahmen eines Arbeitspakets geliefert wird (z. B. Testprotokoll, Architekturdiagramm, Release‑Build).                                                                                                 |
| Definition of Done (DoD)           | Gemeinsame Vereinbarung, welche Kriterien erfüllt sein müssen, damit ein Arbeitspaket oder Backlog‑Eintrag als „fertig“ gilt (z. B. implementiert, getestet, dokumentiert).                                                                   |
| DevOps                             | Kombination aus Praktiken, Tools und Kultur zur engen Verzahnung von Entwicklung (Dev) und Betrieb (Ops), mit dem Ziel, Software schneller, zuverlässiger und wiederholbar auszuliefern (z. B. CI/CD, Infrastruktur als Code, Monitoring).    |
| Epic                               | Größerer Anforderungen‑ bzw. Feature‑Bereich, der in mehrere User Stories zerlegt wird und typischerweise mehrere Sprints umfasst.                                                                                                            |
| FTE (Full Time Equivalent)         | Maß für personelle Kapazität; 1,0 FTE entspricht einer vollen Arbeitskraft (100 %).                                                                                                                                                           |
| Forecast                           | Prognose über zukünftigen Aufwand, Kosten oder Fortschritt basierend auf bisherigen Kennzahlen und Annahmen.                                                                                                                                  |
| Geofencing                         | Ereignisbasierte Standortüberwachung beim Betreten/Verlassen definierter Regionen.                                                                                                                                                            |
| HUD (Head‑Up Display)              | UI‑Element, das kritische Informationen (z. B. Warnungen) prominent und leicht wahrnehmbar darstellt.                                                                                                                                         |
| Meilenstein                        | Wichtiger punktueller Termin bzw. Produktzustand im Projektplan, der ein klar definiertes Ergebnis markiert (z. B. „MVP funktionsfähig“).                                                                                                     |
| MVP (Minimal Viable Product)       | Minimale funktionsfähige Version eines Produkts, die den Kernnutzen bietet und frühzeitig am Markt getestet werden kann.                                                                                                                      |
| Nominatim                          | Geocoding/Reverse‑Geocoding für OSM.                                                                                                                                                                                                          |
| NPV (Net Present Value)            | Kapitalwertverfahren; Barwert aller zukünftigen Cashflows abzüglich der Anfangsinvestition, diskontiert mit einem Zinssatz.                                                                                                                   |
| OAuth                              | Standardisiertes Protokoll zur Autorisierung, mit dem Nutzer Dritten begrenzten Zugriff auf ihre Ressourcen erlauben können, ohne Zugangsdaten direkt weiterzugeben.                                                                          |
| OSM                                | OpenStreetMap – offene Geodatenplattform.                                                                                                                                                                                                     |
| OSM Notes                          | Einfaches Meldesystem in OSM, mit Koordinate + Text.                                                                                                                                                                                          |
| Overpass API                       | Dienst/Abfragesprache für OSM‑Daten (Read‑Only).                                                                                                                                                                                              |
| Owner                              | Verantwortliche Person für ein Thema, Arbeitspaket oder Bereich, die als Ansprechpartner:in und Entscheidungsbefugte:r fungiert.                                                                                                              |
| Payback Period (Amortisationszeit) | Zeitraum, in dem kumulierte Einzahlungen die anfängliche Investition decken; Maß für die Rückflussdauer des eingesetzten Kapitals.                                                                                                            |
| Personentag (PT)                   | Einheit zur Messung von Aufwand; entspricht einem Arbeitstag einer Person (z. B. zur Ermittlung von Sprintkosten).                                                                                                                            |
| POI (Point of Interest)            | Geografischer Punkt von Interesse (z. B. ein gemeldeter Blitzer), der in Karten und Overlays dargestellt und ggf. zwischengespeichert wird.                                                                                                   |
| Product Backlog                    | Zentrale, priorisierte Liste aller Anforderungen, Verbesserungen und Fehlerbehebungen, die im Projekt umgesetzt werden sollen.                                                                                                                |
| Product Owner (PO)                 | Verantwortlich für Produktvision, Priorisierung des Product Backlogs und Abnahme von Ergebnissen im Sprint Review.                                                                                                                            |
| QA (Quality Assurance)             | Maßnahmen und Prozesse zur Sicherstellung der Produktqualität durch Tests, Reviews und kontinuierliche Verbesserung.                                                                                                                          |
| RC (Release Candidate)             | Eine Build‑Version, die potenziell bereit für die Veröffentlichung ist, sofern abschließende Tests und Checks bestanden werden.                                                                                                               |
| Refinement (Backlog Refinement)    | Fortlaufender Prozess, in dem Product Backlog‑Einträge detailliert, geschätzt und priorisiert werden, damit sie für Sprints vorbereitet sind.                                                                                                 |
| Roadmap                            | Zeitliche und strategische Übersicht über geplante Phasen, Meilensteine und Releases eines Produkts über einen längeren Zeitraum.                                                                                                             |
| ROI (Return on Investment)         | Verhältnis von Gewinn (Nettoertrag) zu eingesetztem Kapital; häufig genutzt, um die Wirtschaftlichkeit eines Projekts zu beurteilen.                                                                                                          |
| Scrum Master                       | Unterstützt das Team bei der Anwendung von Scrum, räumt Hindernisse aus dem Weg und fördert kontinuierliche Verbesserung.                                                                                                                     |
| Scrum‑Team                         | Cross‑funktionales Team aus Product Owner, Entwicklerteam und ggf. Scrum Master, das gemeinsam für die Lieferung der Produktinkremente verantwortlich ist.                                                                                    |
| Sprint                             | Feste Iteration mit zwei Wochen Dauer, in der ein Scrum‑Team ein potenziell auslieferbares Inkrement liefert.                                                                                                                                 |
| Sprint Review                      | Treffen am Ende eines Sprints, in dem die erzielten Ergebnisse demonstriert und mit den Stakeholdern besprochen werden.                                                                                                                       |
| Sprint Retrospektive               | Treffen nach dem Sprint Review, in dem das Scrum‑Team seine Zusammenarbeit und den Prozess reflektiert und Verbesserungen für den nächsten Sprint ableitet.                                                                                   |
| Spike                              | Kurzzeitiges, exploratives Arbeitspaket (Experiment/Prototyp), um Unsicherheiten zu reduzieren und technische Entscheidungen vorzubereiten.                                                                                                   |
| Stakeholder                        | Personen oder Gruppen (z. B. Nutzer, Management, Partner), die ein Interesse am Projekt haben oder vom Projektergebnis betroffen sind.                                                                                                        |
| Story Points                       | Relative Maßeinheit zur Schätzung des Aufwands oder der Komplexität von User Stories; dient der Velocity‑Berechnung.                                                                                                                          |
| Tagessatz                          | Kostensatz pro Personentag (€/PT), verwendet zur Multiplikation mit Personentagen für Budget‑Berechnungen.                                                                                                                                    |
| Task (Aufgabe)                     | Kleinste umsetzbare Arbeitseinheit, die aus einer User Story oder einem Epic abgeleitet wird.                                                                                                                                                 |
| TTL (Time To Live)                 | Gültigkeitsdauer von Cache‑Einträgen.                                                                                                                                                                                                         |
| User Story                         | Kurzbeschreibung einer Funktionalität aus Sicht einer Nutzerrolle, üblicherweise formuliert als „Als <Rolle> möchte ich <Ziel>, damit <Nutzen>“. User Stories enthalten meist Akzeptanzkriterien und sind Planungseinheit im Product Backlog. |
| Velocity                           | Kennzahl für den durchschnittlichen Durchsatz eines Teams (z. B. in Story Points) pro Sprint, Grundlage für Forecasts.                                                                                                                        |
| Work Breakdown Structure (WBS)     | Hierarchische Zerlegung des Projekts in Arbeitspakete/Epics/Tasks zur besseren Planung, Steuerung und Zuordnung von Verantwortlichkeiten.                                                                                                     |
